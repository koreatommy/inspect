# 시설정보 사용자화 — 데이터셋 기반 접근 제어 설계

> **문서 기준일:** 2026-05-27 (구조안 A 확정 반영)  
> **목적:** 시스템관리자가 `/admin/upload`에서 업로드한 JSON 파일별로 시설정보를 **데이터셋**으로 구분하고, 사용자별로 할당된 데이터셋에 속한 시설정보만 조회·점검·수정·삭제할 수 있도록 구조를 개선하는 설계 문서  
> **상태:** 분석 완료 / **구조안 A(시설 전역 1행 + 멤버십 M:N + 월점검 `dataset_id`)** 로 §3~§14 정합성 재정렬 / 구현 전

### 구조안 A (확정) — 한 줄 요약

- **`facility_no`(시설번호)는 전역 고유** — `facilities` 행은 DB당 **1건**만 존재한다.  
- **데이터셋 ↔ 시설**은 `facility_dataset_memberships`로 **M:N** — 동일 시설이 여러 데이터셋에 동시에 속할 수 있다.  
- **JSON 업로드**는 대상 데이터셋에 대해: 글로벌 시설·기구·참조섹션 **upsert** + 해당 데이터셋 **멤버십** 갱신·동기화(누락 시 멤버십만 비활성). **타 데이터셋 업로드를 차단하지 않는다**(구 P4 폐기).  
- **`monthly_inspections`**에 `dataset_id`를 두고, **`UNIQUE(facility_no, inspection_month, dataset_id)`** — 데이터셋별로 동일 시설·동일 월 점검을 **각각** 둘 수 있다.

## 1. 현재 구조 분석

### 1.1 시설정보 생성·저장 흐름

```
[관리자] JSON 파일 선택
     ↓
/admin/upload (page.tsx)
     ↓
uploadJsonAction (actions.ts)  ← hasRole(["ADMIN"])
     ↓
uploadFacilityJson (lib/json-parser/uploader.ts)
     ↓ normalizeFacilityPayload → validateFacilityJson → mapFacility
     ↓
Supabase upsert:
  ├── facilities         (시설 기본 → facility_no UNIQUE)
  ├── equipment          (놀이기구 → facility_no + equipment_no UNIQUE)
  ├── facility_legal_inspections   (delete + insert)
  ├── safety_educations            (delete + insert)
  ├── liability_insurances         (delete + insert)
  └── facility_managers            (delete + insert)
```

- JSON 파일은 1개 이상의 시설 정보 배열을 포함합니다.
- `facility_no`를 기준으로 **upsert** 처리하며, 기존 시설이면 업데이트, 없으면 신규 생성합니다.
- **업로드 이력(어떤 파일에서 왔는지, 언제 업로드했는지)은 추적하지 않습니다.**
- JSON 파일명, 업로드 시각, 업로드한 사용자 정보는 어디에도 저장되지 않습니다.

### 1.2 시설정보 테이블 구조 (DB 스키마)

| 테이블 | 용도 | 사용자 필터링 | 비고 |
|--------|------|:----------:|------|
| `facilities` | 시설 기본정보 | ✗ | `facility_no` UNIQUE, `raw_json` 원본 보관 |
| `equipment` | 놀이기구 | ✗ | FK → `facilities(facility_no)` |
| `facility_legal_inspections` | 정기시설검사 | ✗ | FK → `facilities(facility_no)` |
| `safety_educations` | 안전교육 | ✗ | FK → `facilities(facility_no)` |
| `liability_insurances` | 보험 | ✗ | FK → `facilities(facility_no)` |
| `facility_managers` | 관리주체 | ✗ | FK → `facilities(facility_no)` |
| `monthly_inspections` | 월간점검 | ✗ | FK → `facilities(facility_no)`, `created_by` 존재하나 필터 미사용 |
| `monthly_inspection_items` | 점검 항목 | ✗ | FK → `monthly_inspections(id)` |
| `inspection_ledger_rows` | 대장 스냅샷 | ✗ | FK → `monthly_inspections(id)` |

**핵심 문제: `facilities` 테이블에 `dataset_id`나 `source` 컬럼이 없으며, 모든 시설이 하나의 풀에 섞여 있습니다.**

### 1.3 사용자·권한 구조

| 테이블 | 역할 |
|--------|------|
| `auth.users` | Supabase Auth 인증 |
| `inspection_user_roles` | 역할(ADMIN/MANAGER/INSPECTOR/VIEWER), 표시이름, 정지 상태 |

- 사용자와 시설 간 매핑 관계가 **존재하지 않습니다.**
- `inspection_user_roles`에 `dataset_id` 같은 필드가 없습니다.

### 1.4 RLS(Row Level Security) 정책

```sql
-- facilities: 모든 인증 사용자가 전체 시설 조회 가능
CREATE POLICY "Reference facility data is readable by authenticated users"
  ON public.facilities FOR SELECT TO authenticated
  USING (true);   -- ← 필터링 없음

-- monthly_inspections: 동일
CREATE POLICY "Monthly inspections are readable by authenticated users"
  ON public.monthly_inspections FOR SELECT TO authenticated
  USING (true);   -- ← 필터링 없음
```

**모든 SELECT 정책이 `USING (true)`로 설정되어 있어, 인증된 사용자라면 누구나 전체 시설·점검 데이터에 접근할 수 있습니다.**

## 2. 문제점

### 2.1 보안·접근 제어
- **데이터 격리 없음:** MANAGER A가 관리하는 학교 시설을 INSPECTOR B가 모두 열람 가능
- **RLS 수준에서 필터링 없음:** 앱 코드를 우회(예: Supabase REST 직접 호출)하면 전체 데이터 접근 가능
- 사용자가 자신과 무관한 시설에 점검 생성 가능

### 2.2 데이터 관리
- **업로드 이력 미추적:** 어떤 JSON에서 왔는지, 누가 언제 업로드했는지 알 수 없음
- **일괄 교체/삭제 불가:** 특정 업로드 건의 시설만 골라서 제거하거나 재업로드할 수 없음
- 시설 수가 증가하면 전체 목록 조회 성능 저하

### 2.3 확장성
- 향후 **다기관·멀티테넌트** 운영이 불가능
- 사용자별 대시보드 통계가 전체 시설 기준으로만 산출됨
- 보고서 생성 시 범위 지정 불가

## 3. 추천 DB 설계 (구조안 A)

> **의존성:** `facility_datasets` → `facility_dataset_memberships`·`user_dataset_assignments` → `facilities`(기존) / `monthly_inspections`(컬럼 추가).  
> **`facilities.dataset_id`는 사용하지 않는다.** (시설 1행이 단일 데이터셋에 귀속되면 타 데이터셋 사용자의 월점검·조회와 충돌함)

### 3.1 새 테이블: `facility_datasets`

```sql
CREATE TABLE public.facility_datasets (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name        text NOT NULL,                    -- 데이터셋 표시명 (예: "서천군 학교 2026")
  description text,                             -- 설명
  source_file text,                             -- 마지막 업로드 원본 파일명
  uploaded_by uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  facility_count integer NOT NULL DEFAULT 0,  -- 비정규화: 활성 멤버십 기준 시설 수 등
  status      text NOT NULL DEFAULT 'active'   -- CHECK: 'active' | 'archived'
    CHECK (status IN ('active', 'archived')),
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX facility_datasets_status_idx ON public.facility_datasets (status);
CREATE INDEX facility_datasets_created_at_idx ON public.facility_datasets (created_at DESC);
```

### 3.2 새 테이블: `facility_dataset_memberships` (시설 ↔ 데이터셋 M:N)

동일 `facility_no`가 여러 데이터셋에 동시에 속할 수 있게 한다. **RLS·목록·동기화의 기준**은 이 테이블이다.

```sql
CREATE TABLE public.facility_dataset_memberships (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  facility_no  text NOT NULL REFERENCES public.facilities(facility_no)
                 ON UPDATE CASCADE ON DELETE CASCADE,
  dataset_id   uuid NOT NULL REFERENCES public.facility_datasets(id)
                 ON DELETE CASCADE,
  is_active    boolean NOT NULL DEFAULT true,  -- 해당 데이터셋 스냅샷에 포함 여부
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now(),
  UNIQUE (facility_no, dataset_id)
);

CREATE INDEX facility_dataset_memberships_dataset_idx
  ON public.facility_dataset_memberships (dataset_id);
CREATE INDEX facility_dataset_memberships_facility_idx
  ON public.facility_dataset_memberships (facility_no);
CREATE INDEX facility_dataset_memberships_dataset_active_idx
  ON public.facility_dataset_memberships (dataset_id, is_active);
```

### 3.3 `facilities` 테이블 확장 (글로벌 시설 마스터만)

- **`facility_no` UNIQUE 유지** — 전역 1행.
- **추가 권장:** `deactivated_at timestamptz` nullable — 비활성 시각(감사).  
- **`dataset_id` 컬럼은 두지 않는다** (구 설계 폐기).

### 3.4 `monthly_inspections` 확장 (데이터셋별 월점검)

기존 `UNIQUE (facility_no, inspection_month)`는 **전역 1건**이라 데이터셋별 병행 점검과 모순된다.

```sql
ALTER TABLE public.monthly_inspections
  ADD COLUMN dataset_id uuid REFERENCES public.facility_datasets(id)
    ON DELETE RESTRICT;  -- 구현 시: 백필 후 NOT NULL + UNIQUE 교체 순서 준수

-- 기존 UNIQUE 제거 후 (마이그레이션에서 이름 확인)
-- ALTER TABLE public.monthly_inspections DROP CONSTRAINT ...;

-- ALTER TABLE public.monthly_inspections
--   ADD CONSTRAINT monthly_inspections_facility_month_dataset_uniq
--   UNIQUE (facility_no, inspection_month, dataset_id);
```

- **백필:** 기존 행은 모두 **기본 데이터셋** `id`로 `dataset_id` 채움.  
- **신규 점검 생성:** 호출 컨텍스트의 `dataset_id`(예: 사용자가 선택한 활성 데이터셋 또는 할당 1건)를 **필수**로 넣는다.

### 3.5 사용자-데이터셋 매핑: `user_dataset_assignments` (변경 없음)

```sql
CREATE TABLE public.user_dataset_assignments (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  dataset_id  uuid NOT NULL REFERENCES public.facility_datasets(id) ON DELETE CASCADE,
  assigned_by uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, dataset_id)
);

CREATE INDEX user_dataset_assignments_user_idx
  ON public.user_dataset_assignments (user_id);
CREATE INDEX user_dataset_assignments_dataset_idx
  ON public.user_dataset_assignments (dataset_id);
```

### 3.6 ERD 개요 (구조안 A)

```
auth.users
  ├──< inspection_user_roles (역할·상태)
  └──< user_dataset_assignments (사용자 ↔ 데이터셋 M:N)
         │
         ▼
facility_datasets (데이터셋 메타)
  │
  ├──< facility_dataset_memberships (시설 ↔ 데이터셋 M:N, is_active)
  │
  └──< monthly_inspections (facility_no + dataset_id + inspection_month)
         ├──< monthly_inspection_items
         └──< inspection_ledger_rows

facilities (facility_no UNIQUE, 글로벌 마스터 1행)
  ├──< equipment
  ├──< facility_legal_inspections
  ├──< safety_educations
  ├──< liability_insurances
  └──< facility_managers
```

**의존성 주의:** `equipment`·참조 4테이블은 여전히 `facility_no`만 FK로 묶인다. **여러 데이터셋이 같은 시설 행을 공유**하므로, JSON upsert 시 **마지막 업로드가 글로벌 마스터를 덮어쓴다**(§6.5·§12.1 참고).

## 4. 사용자별 데이터셋 매핑 구조

### 4.1 매핑 규칙

| 역할 | 데이터셋 접근 |
|------|-------------|
| **ADMIN** | 전체 데이터셋 (매핑과 무관하게 모든 시설·점검 조회 가능) |
| **MANAGER** | `user_dataset_assignments`에 할당된 `dataset_id`에 연결된 **멤버십·점검**만 |
| **INSPECTOR** | 동일 |
| **VIEWER** | 동일 |

**시설(`facilities`) 조회:** 사용자는 `facility_dataset_memberships`에서  
`(dataset_id ∈ 할당 목록) AND is_active = true` 인 `facility_no`에 대응하는 `facilities` 행만 본다.

**월간점검:** `monthly_inspections.dataset_id ∈ 할당 목록`(또는 ADMIN)으로 필터한다. 동일 `facility_no`·동일 월이라도 **데이터셋이 다르면 별도 행**이 된다.

### 4.2 RLS 헬퍼 함수

```sql
CREATE OR REPLACE FUNCTION app_private.user_accessible_dataset_ids()
RETURNS uuid[]
LANGUAGE plpgsql
STABLE
SECURITY DEFINER
SET search_path = ''
AS $$
DECLARE
  current_role text;
  ids uuid[];
BEGIN
  current_role := app_private.current_user_role();

  -- ADMIN은 전체 접근
  IF current_role = 'ADMIN' THEN
    RETURN NULL;  -- NULL = 제한 없음 (호출부에서 처리)
  END IF;

  SELECT array_agg(dataset_id)
  INTO ids
  FROM public.user_dataset_assignments
  WHERE user_id = auth.uid();

  RETURN COALESCE(ids, '{}');
END;
$$;
```

### 4.3 새 RLS 정책 (`facilities`) — 멤버십 기준

```sql
DROP POLICY IF EXISTS "Reference facility data is readable by authenticated users"
  ON public.facilities;

CREATE POLICY "Users can read facilities via dataset membership"
  ON public.facilities FOR SELECT TO authenticated
  USING (
    app_private.current_user_role() = 'ADMIN'
    OR EXISTS (
      SELECT 1
      FROM public.facility_dataset_memberships m
      JOIN public.user_dataset_assignments uda
        ON uda.dataset_id = m.dataset_id
      WHERE m.facility_no = facilities.facility_no
        AND m.is_active = true
        AND uda.user_id = auth.uid()
    )
  );
```

### 4.4 `equipment` 및 참조 4테이블 RLS

시설과 동일하게 **`facility_no`가 사용자에게 노출된 시설**에 한해 접근한다.

```sql
-- 예: equipment SELECT
USING (
  app_private.current_user_role() = 'ADMIN'
  OR EXISTS (
    SELECT 1 FROM public.facility_dataset_memberships m
    JOIN public.user_dataset_assignments uda ON uda.dataset_id = m.dataset_id
    WHERE m.facility_no = equipment.facility_no
      AND m.is_active = true
      AND uda.user_id = auth.uid()
  )
);
```

> `facility_legal_inspections`, `safety_educations`, `liability_insurances`, `facility_managers`에도 동일 패턴을 적용한다.

### 4.5 `monthly_inspections` RLS — `dataset_id` 직접 필터

```sql
DROP POLICY IF EXISTS "Monthly inspections are readable by authenticated users"
  ON public.monthly_inspections;

CREATE POLICY "Users can read inspections for assigned datasets"
  ON public.monthly_inspections FOR SELECT TO authenticated
  USING (
    app_private.current_user_role() = 'ADMIN'
    OR dataset_id IN (
      SELECT dataset_id
      FROM public.user_dataset_assignments
      WHERE user_id = auth.uid()
    )
  );
```

INSERT/UPDATE는 동일하게 `dataset_id ∈ 할당` (또는 ADMIN)을 검증한다.

### 4.6 `monthly_inspection_items`, `inspection_ledger_rows`

부모 `monthly_inspections`의 `dataset_id`·접근 정책을 따른다(간접 필터 또는 `inspection_id` 서브쿼리).

## 6. API/서버 로직 수정 방향

### 6.1 핵심 원칙: RLS 우선, 앱 코드 보조

RLS 기반 필터링이 적용되면 **대부분의 기존 쿼리 코드를 수정할 필요가 없습니다.** Supabase 클라이언트가 인증 사용자의 JWT로 요청하므로, RLS가 자동으로 `user_dataset_assignments`·`facility_dataset_memberships`를 기준으로 필터링합니다.

**예외:** 월간점검 생성 시 **`dataset_id`를 insert에 넣는 코드**는 반드시 추가한다(§3.4).

### 6.2 업로드 흐름 변경 (구조안 A)

```
[관리자] JSON 파일 + 대상 데이터셋 선택(신규 생성 또는 기존 active)
     ↓
uploadJsonAction
     ↓
1. facility_datasets INSERT 또는 기존 id 선택 (status = active 확인)
2. uploadFacilityJson(supabase, payload, { datasetId })
     ↓
3. 각 시설: facilities UPSERT (글로벌 마스터, facility_no UNIQUE)
4. facility_dataset_memberships UPSERT (해당 datasetId, is_active=true)
5. equipment·참조 섹션 동기화 (기존 로직, 시설 단위)
6. 동일 datasetId에 한해 멤버십 동기화: JSON 미포함 시설 → 멤버십 is_active=false (§6.5)
7. facility_datasets 메타 갱신 (facility_count, source_file, uploaded_by, updated_at)
```

**기존 업로드와의 차이:**
- `facilities`에 **`dataset_id`를 쓰지 않는다.**
- 동일 `facility_no`가 **다른 데이터셋 JSON에 동시에 있어도 허용** — 각 데이터셋에 **멤버십 행**만 추가/갱신한다.
- 글로벌 `facilities`·`equipment`·참조 4종은 **마지막 업로드가 덮어쓴다**(§6.5 트레이드오프).

### 6.3 데이터셋 사용자 할당

- **관리자 화면 (`/settings/users`):** 사용자 생성·수정 시 데이터셋 체크박스로 선택
- **데이터셋 관리 화면 (`/admin/datasets`):** (선택) 데이터셋별 할당된 사용자·멤버십 시설 목록
- 양방향 관리 지원 (사용자 → 데이터셋, 데이터셋 → 사용자)

### 6.4 대시보드 통계 자동 필터링

`dashboard-content.tsx`의 기존 쿼리:

```typescript
supabase.from("facilities").select("*", { count: "exact", head: true })
```

RLS 적용 후 이 쿼리는 **사용자에게 노출되는 시설(활성 멤버십 경유)**만 카운트합니다. 코드 변경 없이 자동 적용됩니다.

### 6.5 JSON 동기화·`is_active` 정책 (구조안 A)

PRD 8.3(기구) 및 데이터셋 스냅샷을 **멤버십**으로 표현한 규칙이다.

#### 동기화 범위

| 대상 | JSON에 있음 (대상 `dataset_id` 업로드 시) | JSON에 없음 (**동일 `dataset_id` 멤버십**만) | 다른 `dataset_id` |
|------|-------------|---------------------------------------------|-------------------|
| **시설 마스터** (`facilities`) | upsert, 글로벌 1행 갱신 | 변경 없음(※) | 변경 없음 |
| **데이터셋 멤버십** | upsert `is_active=true` | **`is_active=false`** (행 삭제 금지 권장) | 변경 없음 |
| **기구** (`equipment[]`) | upsert, `is_active=true` | JSON에 없는 `rideSn` → `is_active=false` | 변경 없음 |
| **월간 점검·대장** | — | **변경 없음** (물리 삭제 금지) | RLS로 접근 제어 |
| **참조 섹션** | delete + insert (시설별, 글로벌) | — | — |

(※) 시설 **전역** 비활성(`facilities.is_active`)이 필요하면 별도 정책으로 “**활성 멤버십 0건**일 때만 false” 등을 정한다(1차 선택).

#### 업로드 1회 처리 흐름 (Phase 3 이후 목표)

```text
업로드 대상 dataset_id = T
업로드 JSON의 pfctSn 집합 = S
멤버십 상 T에 속한 facility_no 집합 (처리 전 스냅샷) = M

• 각 pfctSn ∈ S
    → facilities UPSERT (글로벌)
    → memberships UPSERT (facility_no, T, is_active=true)
    → equipment·참조 섹션 동기화 (해당 시설 JSON 기준, 기존 uploader)
• M \ S  → UPDATE memberships SET is_active=false WHERE dataset_id=T AND facility_no ∈ (M\S)
• 다른 dataset_id의 멤버십·시설 마스터 → 위 동기화로 **의도적으로 간접 영향** 가능(기구·참조는 공유). 운영 가이드 필요.
```

- JSON에 **다시 포함**되면 해당 데이터셋에 대해 멤버십 `is_active=true`로 복구.
- **`dataset_id` 없이** “전체 DB 대비 JSON 누락 시설 비활성화”는 하지 않는다.

#### `facility_no` 전역 UNIQUE (유지)

- 한 `facility_no`는 `facilities`에 **한 행**만 존재한다.
- **다른 데이터셋 JSON**에 동일 `pfctSn`이 오면 **INSERT가 아니라 upsert**이며, **해당 데이터셋에 대한 멤버십**을 추가하거나 활성화한다. **업로드 차단(구 P4)은 하지 않는다.**

#### 글로벌 마스터 공유의 트레이드오프

- `basic`·`equipment`·참조 4종은 **시설당 1벌**이므로, 데이터셋 A와 B가 같은 시설을 공유하면 **나중에 업로드한 JSON이 마스터를 덮어쓴다.**
- 완화책(향후): 업로드 시 경고 로그, `last_upload_dataset_id` 기록, 기관별 확장 스키마 분리 등.

#### Phase 0 (선행)과의 경계

| 항목 | Phase 0 | Phase 3+ (구조안 A) |
|------|---------|---------------------|
| 비활성 시설 **조회 UI** | ✅ 데이터 관리 하위 메뉴 | **멤버십**·데이터셋 필터 |
| 업로드 시 **스냅샷 반영** | ⏸ | ✅ 대상 `dataset_id`의 **멤버십**만 `is_active` 동기화 |
| `uploadFacilityJson` 시그니처 | `options?: { datasetId?: string }` 타입·주석 | `{ datasetId }` 필수 |

## 9. 기존 데이터 마이그레이션 전략 (구조안 A)

### 9.1 필요성

필요합니다. 기존 데이터는 `facility_dataset_memberships`, `monthly_inspections.dataset_id`가 없어서 RLS 기준을 만들 수 없습니다.

### 9.2 마이그레이션 절차

```sql
-- 1) 기본 데이터셋 생성
INSERT INTO public.facility_datasets (id, name, description, status, facility_count)
VALUES (
  gen_random_uuid(),
  '기존 시설 데이터',
  '마이그레이션: 구조안 A 기본 데이터셋',
  'active',
  (SELECT count(*) FROM public.facilities WHERE is_active = true)
);

-- 2) 기존 시설 -> 기본 데이터셋 멤버십 생성 (활성)
INSERT INTO public.facility_dataset_memberships (facility_no, dataset_id, is_active)
SELECT f.facility_no, fd.id, COALESCE(f.is_active, true)
FROM public.facilities f
CROSS JOIN public.facility_datasets fd
WHERE fd.name = '기존 시설 데이터'
ON CONFLICT (facility_no, dataset_id) DO NOTHING;

-- 3) 비-ADMIN 사용자 기본 데이터셋 할당
INSERT INTO public.user_dataset_assignments (user_id, dataset_id)
SELECT ur.user_id, fd.id
FROM public.inspection_user_roles ur
CROSS JOIN public.facility_datasets fd
WHERE fd.name = '기존 시설 데이터'
  AND ur.role != 'ADMIN'
ON CONFLICT (user_id, dataset_id) DO NOTHING;

-- 4) monthly_inspections.dataset_id 백필
UPDATE public.monthly_inspections mi
SET dataset_id = fd.id
FROM public.facility_datasets fd
WHERE fd.name = '기존 시설 데이터'
  AND mi.dataset_id IS NULL;
```

### 9.3 안전성 체크

- `facility_dataset_memberships` 누락 0건
- `monthly_inspections.dataset_id IS NULL` 0건
- 기존 점검 건수 불변
- `UNIQUE(facility_no, inspection_month)`를 `(facility_no, inspection_month, dataset_id)`로 교체 전 중복 데이터 점검

## 11. 정책 결정사항 (구조안 A)

| ID | 결정 |
|----|------|
| P1 | `facility_no` 전역 UNIQUE 유지 (`facilities` 1행) |
| P2 | 데이터셋-시설 관계는 `facility_dataset_memberships` M:N |
| P3 | 동일 `facility_no`가 여러 데이터셋 JSON에 등장 가능 (차단하지 않음) |
| P4 | `monthly_inspections`는 `dataset_id` 포함 유니크로 분리 관리 |
| P5 | 데이터셋 삭제는 hard delete 대신 `status='archived'` |
| P6 | RLS는 마지막 단계에서 적용 |

## 12. 운영 시 유의사항

### 12.1 글로벌 마스터 공유 충돌

동일 시설번호를 여러 데이터셋이 공유하므로, 시설 기본정보·기구·참조 섹션은 마지막 업로드 기준으로 갱신된다. 기관별 필드 차이가 큰 경우 별도 확장 테이블이 필요하다.

### 12.2 월간점검 이력 보존

- `monthly_inspections`, `monthly_inspection_items`, `inspection_ledger_rows` 물리 삭제 금지
- 비활성 멤버십이어도 기존 점검 조회는 정책에 따라 허용 가능

### 12.3 archived 정책

| 동작 | 허용 |
|------|:----:|
| JSON 업로드 | ✗ |
| 신규 월간점검 생성 | ✗ |
| 기존 점검 조회 | ✓ (할당 또는 ADMIN) |

## 13. 최종 권장안 요약

구조안 A는 `facility_no`의 공공 고유성을 유지하면서도, 사용자별 데이터셋 접근과 데이터셋별 월간점검 독립성을 동시에 만족합니다. 핵심은 `facility_dataset_memberships`(조회 경계)와 `monthly_inspections.dataset_id`(점검 경계)를 분리해 관리하는 것입니다.
