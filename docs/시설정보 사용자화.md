# 시설정보 사용자화 — 데이터셋 기반 접근 제어 설계

> **문서 기준일:** 2026-05-26 (Phase 0·결정 사항 보강)  
> **목적:** 시스템관리자가 `/admin/upload`에서 업로드한 JSON 파일별로 시설정보를 **데이터셋**으로 구분하고, 사용자별로 할당된 데이터셋에 속한 시설정보만 조회·점검·수정·삭제할 수 있도록 구조를 개선하는 설계 문서  
> **상태:** 분석 완료 / **§14 1차 구현 개발 플랜 확정** / 구현 전

## 1. 현재 구조 분석

### 1.1 시설정보 생성·저장 흐름

```
[관리자] JSON 파일 선택
     ↓
/admin/upload (page.tsx)
     ↓
uploadJsonAction (actions.ts)  ← hasRole(["ADMIN"])
     ↓
uploadFacilityJson (lib/json-parser/uploader.ts)
     ↓ normalizeFacilityPayload → validateFacilityJson → mapFacility
     ↓
Supabase upsert:
  ├── facilities         (시설 기본 → facility_no UNIQUE)
  ├── equipment          (놀이기구 → facility_no + equipment_no UNIQUE)
  ├── facility_legal_inspections   (delete + insert)
  ├── safety_educations            (delete + insert)
  ├── liability_insurances         (delete + insert)
  └── facility_managers            (delete + insert)
```

- JSON 파일은 1개 이상의 시설 정보 배열을 포함합니다.
- `facility_no`를 기준으로 **upsert** 처리하며, 기존 시설이면 업데이트, 없으면 신규 생성합니다.
- **업로드 이력(어떤 파일에서 왔는지, 언제 업로드했는지)은 추적하지 않습니다.**
- JSON 파일명, 업로드 시각, 업로드한 사용자 정보는 어디에도 저장되지 않습니다.

### 1.2 시설정보 테이블 구조 (DB 스키마)

| 테이블 | 용도 | 사용자 필터링 | 비고 |
|--------|------|:----------:|------|
| `facilities` | 시설 기본정보 | ✗ | `facility_no` UNIQUE, `raw_json` 원본 보관 |
| `equipment` | 놀이기구 | ✗ | FK → `facilities(facility_no)` |
| `facility_legal_inspections` | 정기시설검사 | ✗ | FK → `facilities(facility_no)` |
| `safety_educations` | 안전교육 | ✗ | FK → `facilities(facility_no)` |
| `liability_insurances` | 보험 | ✗ | FK → `facilities(facility_no)` |
| `facility_managers` | 관리주체 | ✗ | FK → `facilities(facility_no)` |
| `monthly_inspections` | 월간점검 | ✗ | FK → `facilities(facility_no)`, `created_by` 존재하나 필터 미사용 |
| `monthly_inspection_items` | 점검 항목 | ✗ | FK → `monthly_inspections(id)` |
| `inspection_ledger_rows` | 대장 스냅샷 | ✗ | FK → `monthly_inspections(id)` |

**핵심 문제: `facilities` 테이블에 `dataset_id`나 `source` 컬럼이 없으며, 모든 시설이 하나의 풀에 섞여 있습니다.**

### 1.3 사용자·권한 구조

| 테이블 | 역할 |
|--------|------|
| `auth.users` | Supabase Auth 인증 |
| `inspection_user_roles` | 역할(ADMIN/MANAGER/INSPECTOR/VIEWER), 표시이름, 정지 상태 |

- 사용자와 시설 간 매핑 관계가 **존재하지 않습니다.**
- `inspection_user_roles`에 `dataset_id` 같은 필드가 없습니다.

### 1.4 RLS(Row Level Security) 정책

```sql
-- facilities: 모든 인증 사용자가 전체 시설 조회 가능
CREATE POLICY "Reference facility data is readable by authenticated users"
  ON public.facilities FOR SELECT TO authenticated
  USING (true);   -- ← 필터링 없음

-- monthly_inspections: 동일
CREATE POLICY "Monthly inspections are readable by authenticated users"
  ON public.monthly_inspections FOR SELECT TO authenticated
  USING (true);   -- ← 필터링 없음
```

**모든 SELECT 정책이 `USING (true)`로 설정되어 있어, 인증된 사용자라면 누구나 전체 시설·점검 데이터에 접근할 수 있습니다.**

## 2. 문제점

### 2.1 보안·접근 제어
- **데이터 격리 없음:** MANAGER A가 관리하는 학교 시설을 INSPECTOR B가 모두 열람 가능
- **RLS 수준에서 필터링 없음:** 앱 코드를 우회(예: Supabase REST 직접 호출)하면 전체 데이터 접근 가능
- 사용자가 자신과 무관한 시설에 점검 생성 가능

### 2.2 데이터 관리
- **업로드 이력 미추적:** 어떤 JSON에서 왔는지, 누가 언제 업로드했는지 알 수 없음
- **일괄 교체/삭제 불가:** 특정 업로드 건의 시설만 골라서 제거하거나 재업로드할 수 없음
- 시설 수가 증가하면 전체 목록 조회 성능 저하

### 2.3 확장성
- 향후 **다기관·멀티테넌트** 운영이 불가능
- 사용자별 대시보드 통계가 전체 시설 기준으로만 산출됨
- 보고서 생성 시 범위 지정 불가

## 3. 추천 DB 설계

### 3.1 새 테이블: `facility_datasets`

```sql
CREATE TABLE public.facility_datasets (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name        text NOT NULL,                    -- 데이터셋 표시명 (예: "서천군 학교 2026")
  description text,                             -- 설명
  source_file text,                             -- 업로드 원본 파일명
  uploaded_by uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  facility_count integer NOT NULL DEFAULT 0,    -- 비정규화: 포함 시설 수
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);
```

### 3.2 `facilities` 테이블 확장

```sql
ALTER TABLE public.facilities
  ADD COLUMN dataset_id uuid REFERENCES public.facility_datasets(id)
    ON DELETE SET NULL;

CREATE INDEX facilities_dataset_id_idx ON public.facilities (dataset_id);
```

### 3.3 사용자-데이터셋 매핑 테이블: `user_dataset_assignments`

```sql
CREATE TABLE public.user_dataset_assignments (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  dataset_id  uuid NOT NULL REFERENCES public.facility_datasets(id) ON DELETE CASCADE,
  assigned_by uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, dataset_id)
);

CREATE INDEX user_dataset_assignments_user_idx
  ON public.user_dataset_assignments (user_id);
CREATE INDEX user_dataset_assignments_dataset_idx
  ON public.user_dataset_assignments (dataset_id);
```

### 3.4 ERD 개요

```
auth.users
  ├──< inspection_user_roles (역할·상태)
  └──< user_dataset_assignments (사용자↔데이터셋 M:N)
         │
         ▼
facility_datasets (데이터셋 메타)
  │
  ▼
facilities (dataset_id FK)
  ├──< equipment
  ├──< facility_legal_inspections
  ├──< safety_educations
  ├──< liability_insurances
  ├──< facility_managers
  └──< monthly_inspections
         ├──< monthly_inspection_items
         └──< inspection_ledger_rows
```

## 4. 사용자별 데이터셋 매핑 구조

### 4.1 매핑 규칙

| 역할 | 데이터셋 접근 |
|------|-------------|
| **ADMIN** | 전체 데이터셋 (매핑과 무관하게 모든 시설 접근) |
| **MANAGER** | `user_dataset_assignments`에 할당된 데이터셋만 |
| **INSPECTOR** | `user_dataset_assignments`에 할당된 데이터셋만 |
| **VIEWER** | `user_dataset_assignments`에 할당된 데이터셋만 |

### 4.2 RLS 헬퍼 함수

```sql
CREATE OR REPLACE FUNCTION app_private.user_accessible_dataset_ids()
RETURNS uuid[]
LANGUAGE plpgsql
STABLE
SECURITY DEFINER
SET search_path = ''
AS $$
DECLARE
  current_role text;
  ids uuid[];
BEGIN
  current_role := app_private.current_user_role();

  -- ADMIN은 전체 접근
  IF current_role = 'ADMIN' THEN
    RETURN NULL;  -- NULL = 제한 없음 (호출부에서 처리)
  END IF;

  SELECT array_agg(dataset_id)
  INTO ids
  FROM public.user_dataset_assignments
  WHERE user_id = auth.uid();

  RETURN COALESCE(ids, '{}');
END;
$$;
```

### 4.3 새 RLS 정책 (facilities)

```sql
-- 기존 정책 교체
DROP POLICY "Reference facility data is readable by authenticated users"
  ON public.facilities;

CREATE POLICY "Users can read assigned facilities"
  ON public.facilities FOR SELECT TO authenticated
  USING (
    app_private.current_user_role() = 'ADMIN'
    OR dataset_id IN (
      SELECT dataset_id
      FROM public.user_dataset_assignments
      WHERE user_id = auth.uid()
    )
  );
```

### 4.4 monthly_inspections RLS 확장

```sql
DROP POLICY "Monthly inspections are readable by authenticated users"
  ON public.monthly_inspections;

CREATE POLICY "Users can read inspections for assigned facilities"
  ON public.monthly_inspections FOR SELECT TO authenticated
  USING (
    app_private.current_user_role() = 'ADMIN'
    OR facility_no IN (
      SELECT f.facility_no
      FROM public.facilities f
      JOIN public.user_dataset_assignments uda
        ON uda.dataset_id = f.dataset_id
      WHERE uda.user_id = auth.uid()
    )
  );
```

> 같은 패턴으로 `equipment`, `facility_legal_inspections`, `safety_educations`, `liability_insurances`, `facility_managers`, `monthly_inspection_items`, `inspection_ledger_rows`에도 적용합니다.

## 6. API/서버 로직 수정 방향

### 6.1 핵심 원칙: RLS 우선, 앱 코드 보조

RLS 기반 필터링이 적용되면 **대부분의 기존 쿼리 코드를 수정할 필요가 없습니다.** Supabase 클라이언트가 인증 사용자의 JWT로 요청하므로, RLS가 자동으로 `user_dataset_assignments`를 기준으로 필터링합니다.

### 6.2 업로드 흐름 변경

```
[관리자] JSON 파일 + 데이터셋 이름 입력
     ↓
uploadJsonAction
     ↓
1. facility_datasets INSERT (name, source_file, uploaded_by)  → dataset_id
2. uploadFacilityJson(supabase, payload, dataset_id)
     ↓
3. facilities UPSERT 시 dataset_id 포함
     ↓
4. facility_datasets.facility_count UPDATE
```

**기존 업로드와의 차이:**
- 데이터셋 이름 필수 입력
- `facility_datasets` 레코드 먼저 생성
- `facilities` upsert 시 `dataset_id` 컬럼 포함
- 기존 시설의 데이터셋이 변경될 수 있음 (동일 `facility_no` 재업로드 시)

### 6.3 데이터셋 사용자 할당

- **관리자 화면 (`/settings/users`):** 사용자 생성·수정 시 데이터셋 체크박스로 선택
- **데이터셋 관리 화면 (`/admin/datasets`):** 데이터셋별로 할당된 사용자 목록 관리
- 양방향 관리 지원 (사용자 → 데이터셋, 데이터셋 → 사용자)

### 6.4 대시보드 통계 자동 필터링

`dashboard-content.tsx`의 기존 쿼리:

```typescript
supabase.from("facilities").select("*", { count: "exact", head: true })
```

RLS 적용 후 이 쿼리는 **사용자에게 할당된 데이터셋의 시설만 카운트**합니다. 코드 변경 없이 자동 적용됩니다.

### 6.5 JSON 동기화·`is_active` 정책 (시설·기구)

PRD 8.3(기구) 및 운영 요구(시설)를 데이터셋 단위로 통합한 규칙입니다.

#### 동기화 범위

| 대상 | JSON에 있음 | JSON에 없음 (동일 `dataset_id` 소속) | 다른 `dataset_id` |
|------|-------------|--------------------------------------|-------------------|
| **시설** (`basic.pfctSn`) | upsert, `is_active = true` | **`is_active = false`** (DELETE 금지) | 변경 없음 |
| **기구** (`equipment[].rideSn`) | upsert, `is_active = true` | **`is_active = false`** (DELETE 금지) | (시설 소속 따라 간접) |
| **월간 점검·대장** | — | **변경 없음** (물리 삭제 금지) | RLS로 접근 제어 (Phase 6) |
| **참조 섹션** (법정검사·교육·보험·관리주체) | delete + insert (시설별) | — | — |

#### 업로드 1회 처리 흐름 (Phase 3 이후 목표)

```text
업로드 JSON의 pfctSn 집합 = S
해당 dataset_id의 DB 시설 집합 (is_active 무관) = D

• S ∩ D  → facilities upsert + is_active = true, equipment 동기화
• S \ D  → 신규 INSERT(upsert) + dataset_id 설정
• D \ S  → UPDATE is_active = false  (행 삭제 없음)
• 다른 dataset_id 소속 시설 → 일체 변경 없음
```

- JSON에 **다시 포함**되면 `is_active = true`로 복구 (기구와 동일).
- **`dataset_id` 없이** “전체 DB 대비 JSON 누락 시설 비활성화”는 하지 않는다. (다른 데이터셋 오염 방지 — §12.1)

#### `facility_no` 전역 UNIQUE

- 한 `facility_no`는 DB에 **한 행**만 존재한다.
- 다른 데이터셋 JSON에 동일 `pfctSn`이 오면 **INSERT가 아니라 upsert**이며, 정책에 따라 `dataset_id` **이전**이 발생할 수 있다 (§12.1 D3에서 결정).

#### Phase 0 (선행)과의 경계

| 항목 | Phase 0 | Phase 3+ (데이터셋 연동 후) |
|------|---------|------------------------------|
| 비활성 시설 **조회 UI** | ✅ 데이터 관리 하위 메뉴 | 데이터셋 필터 추가 |
| 업로드 시 시설 **자동 비활성화** | ⏸ `dataset_id` 도입과 **동시** 권장 | ✅ `dataset_id` 범위만 |
| `uploadFacilityJson` 시그니처 | `options?: { datasetId?: string }` 자리만 | `datasetId` 필수 |

ADMIN은 모든 데이터셋에 접근할 수 있어야 합니다. RLS 정책에서 `current_user_role() = 'ADMIN'` 조건으로 바이패스합니다. 이는 현재 시스템의 ADMIN 전체 접근 정책과 일관됩니다.

### 8.4 데이터셋 미할당 사용자

데이터셋이 할당되지 않은 MANAGER/INSPECTOR/VIEWER는 **시설 0건**이 보이게 됩니다. 이는 의도된 동작입니다. 관리자가 사용자를 생성한 후 데이터셋을 할당해야 합니다.

### 8.5 성능 고려

| 항목 | 대응 |
|------|------|
| RLS 서브쿼리 성능 | `user_dataset_assignments`에 `(user_id)` 인덱스 + `facilities`에 `(dataset_id)` 인덱스 |
| 다중 데이터셋 사용자 | `IN` 서브쿼리로 처리, 데이터셋 수가 소규모(~수십)이므로 성능 영향 미미 |
| 기존 쿼리 | RLS 자동 적용, 추가 JOIN 발생하나 인덱스로 대응 |

## 9. 기존 데이터 마이그레이션 전략

### 9.1 마이그레이션 필요 여부

**필요합니다.** 기존 60건의 시설 데이터에 `dataset_id`가 없으므로 기본 데이터셋을 생성하고 할당해야 합니다.

### 9.2 마이그레이션 절차

```sql
-- 1. 기본 데이터셋 생성
INSERT INTO public.facility_datasets (id, name, description, facility_count)
VALUES (
  gen_random_uuid(),
  '기존 시설 데이터',
  '마이그레이션: 데이터셋 도입 전 기존 시설',
  (SELECT count(*) FROM public.facilities)
);

-- 2. 모든 기존 시설에 기본 데이터셋 할당
UPDATE public.facilities
SET dataset_id = (SELECT id FROM public.facility_datasets WHERE name = '기존 시설 데이터')
WHERE dataset_id IS NULL;

-- 3. 기존 ADMIN이 아닌 사용자에게 기본 데이터셋 할당
INSERT INTO public.user_dataset_assignments (user_id, dataset_id)
SELECT ur.user_id, fd.id
FROM public.inspection_user_roles ur
CROSS JOIN public.facility_datasets fd
WHERE fd.name = '기존 시설 데이터'
  AND ur.role != 'ADMIN'
ON CONFLICT (user_id, dataset_id) DO NOTHING;
```

### 9.3 마이그레이션 안전성

- **비파괴적:** 기존 데이터를 삭제하지 않고 `dataset_id` 컬럼만 추가
- **점진적:** 새 RLS 정책 적용 전에 모든 시설에 dataset_id 할당 확인
- **롤백 가능:** `dataset_id` 컬럼 DROP + 새 테이블 DROP + 기존 RLS 복원

### 9.4 `dataset_id` NULL 처리

신규 RLS에서 `dataset_id IS NULL`인 시설은 **ADMIN만 볼 수 있도록** 처리합니다. 이는 마이그레이션 누락이나 잘못된 업로드를 방지합니다.

```sql
USING (
  app_private.current_user_role() = 'ADMIN'
  OR (
    dataset_id IS NOT NULL
    AND dataset_id IN (
      SELECT dataset_id FROM public.user_dataset_assignments
      WHERE user_id = auth.uid()
    )
  )
);
```

## 11. Phase 0 상세 — 비활성 시설 관리 (선행 작업)

### 11.1 목적

- JSON 재업로드로 목록에서 빠진 시설을 **삭제하지 않고** `is_active = false`로 남긴 뒤, 시스템 관리자가 **데이터 관리 → 비활성 시설**에서 확인한다.
- 이후 **다른 JSON → 다른 데이터셋**·사용자 할당(Phase 2~6)과 충돌하지 않도록, 비활성화 범위를 **전역이 아닌 `dataset_id`**로 한정할 준비만 한다.

### 11.2 현재 코드와의 차이 (2026-05 기준)

| 동작 | 현재 | Phase 0 목표 | Phase 3 목표 |
|------|------|--------------|--------------|
| JSON에 없는 **기구** | `is_active = false` | 동일 | 동일 (`dataset` 무관, 시설 단위) |
| JSON에 없는 **시설** | **무변경** | UI만 (목록) / 동기화는 Phase 3 | `dataset_id` 범위 비활성화 |
| JSON에 없는 시설 (DB만 존재) | 목록에 계속 표시 | 비활성 메뉴에서만 | + 데이터셋 필터 |

### 11.3 구현 시 유의 (Phase 2와의 접점)

- 비활성 목록 쿼리를 **한 모듈**로 두어 Phase 3에서 `WHERE dataset_id = $1` 한 줄 추가.
- 인덱스 권장: `(is_active)`, Phase 1 이후 `(dataset_id, is_active)`.
- **전역** “JSON에 없으면 모든 `facilities` 비활성” 로직은 **금지** (§12.1).

| D2 | 같은 기관 **월간 갱신** | A) 매번 신규 데이터셋 · B) 동일 `dataset_id`에 merge | “비활성 시설”이 어느 업로드 건 기준인지 |
| D3 | 동일 `pfctSn`이 **다른 데이터셋** JSON에 등장 | A) 거부 · B) upsert + `dataset_id` **이전** · C) 관리자 확인 후 이전 | 타 기관 데이터 오염 여부 |
| D4 | **데이터셋 삭제** | A) 시설 `dataset_id` NULL · B) 시설도 비활성 · C) 데이터셋만 soft-delete | RLS·고아 시설·NULL 정책 (§9.4) |
| D5 | `facility_datasets.facility_count` | 실시간 count vs 비정규화 유지 | 대시보드·목록 정합성 |

**권장 초안:** 운영이 “기관별 정기 JSON 갱신”이면 **D1=B, D2=B**를 기본으로 검토. “파일별 스냅샷 보관”이면 **D1=A**.

## 13. 최종 권장안

### 13.1 구조 선택: 별도 데이터셋 테이블 (권장)

| 방식 | 장점 | 단점 |
|------|------|------|
| **A. JSON 파일명만 `facilities`에 저장** | 구현 간단 | 사용자 매핑 불가, 데이터셋 메타관리 불가, 확장성 없음 |
| **B. 별도 `facility_datasets` 테이블 (권장)** | 메타 관리, 사용자 매핑, 이력 추적, 확장 용이 | 초기 구현 비용 다소 높음 |

**권장안: B안 (별도 데이터셋 테이블)**

이유:
- 사용자별 시설 접근 제어가 핵심 요구사항이므로 `user_dataset_assignments` 매핑 테이블이 필수
- JSON 파일명만으로는 사용자-시설 매핑을 표현할 수 없음
- 데이터셋 단위의 일괄 관리(삭제, 재업로드, 통계)가 가능해짐
- 향후 멀티테넌트·다기관 운영으로 확장 시 데이터셋 개념이 그대로 활용 가능

### 13.2 RLS 우선 전략

**RLS에서 필터링하면 앱 코드 변경이 최소화됩니다.**

- 기존 15+ 파일의 시설/점검 쿼리 코드를 수정할 필요 없이, RLS가 자동으로 사용자의 데이터셋에 맞는 데이터만 반환합니다.
- 앱 코드에서는 **업로드·관리자 화면 신규 구현**만 필요합니다.
- Supabase REST API 직접 접근 시에도 RLS가 작동하여 보안이 유지됩니다.

### 13.3 향후 확장 포인트

- **보고서 생성:** 데이터셋 단위로 보고서 범위 자동 지정
- **대시보드 개선:** 데이터셋별 통계 탭/필터
- **감사 로그:** 데이터셋 변경 이력 추적
- **API 확장:** 외부 시스템 연동 시 데이터셋 기반 접근 토큰 발급
- **다기관 운영:** 데이터셋을 기관(organization) 개념으로 확장

### 14.3 기존 데이터 — 기본 데이터셋 연결

```sql
-- 1) 기본 데이터셋
INSERT INTO public.facility_datasets (name, description, status, facility_count)
VALUES (
  '기존 시설 데이터',
  '데이터셋 도입 전 마이그레이션',
  'active',
  (SELECT count(*) FROM public.facilities WHERE is_active = true)
);

-- 2) 시설 연결
UPDATE public.facilities
SET dataset_id = (
  SELECT id FROM public.facility_datasets WHERE name = '기존 시설 데이터' LIMIT 1
)
WHERE dataset_id IS NULL;

-- 3) 비-ADMIN 사용자 할당 (ADMIN은 RLS 바이패스, 행 불필요)
INSERT INTO public.user_dataset_assignments (user_id, dataset_id)
SELECT ur.user_id, fd.id
FROM public.inspection_user_roles ur
CROSS JOIN public.facility_datasets fd
WHERE fd.name = '기존 시설 데이터'
  AND ur.role != 'ADMIN'
ON CONFLICT (user_id, dataset_id) DO NOTHING;

-- 4) facility_count 재계산
UPDATE public.facility_datasets fd
SET facility_count = (
  SELECT count(*) FROM public.facilities f
  WHERE f.dataset_id = fd.id AND f.is_active = true
);
```

**검증 체크:**

- `facilities.dataset_id IS NULL` → 0건
- 운영 역할(MANAGER/INSPECTOR/VIEWER) 중 할당 0건 사용자 목록 확인
- 기존 `monthly_inspections` 건수 불변

### 14.4 JSON 업로드 동기화 로직

#### 입력

- 대상 `dataset_id` (`status = 'active'` 만)
- JSON payload (배열 / `details` / 단일 객체 — 기존과 동일)

#### 처리 순서 (데이터셋 단위, 트랜잭션 권장)

```text
1. dataset.status = 'active' 확인 (archived → 거부)
2. payload 검증 (기존 validator)
3. 충돌 검사 (P4)
   facilities.facility_no ∈ payload.pfctSn
   AND dataset_id IS NOT NULL
   AND dataset_id <> :targetDatasetId
   → 1건이라도 있으면 전체 중단 + 충돌 facility_no 목록 반환
4. 각 시설 uploadFacility(json, datasetId):
   a. mapFacility + dataset_id, is_active=true, deactivated_at=NULL
   b. facilities UPSERT ON CONFLICT (facility_no)
   c. equipment 동기화 (JSON에 없는 rideSn → is_active=false)
   d. 참조 섹션 delete + insert (기존)
5. 시설 비활성화 (동일 dataset_id만)
   D = { facility_no | dataset_id = :target }
   S = { pfctSn from payload }
   D \ S → is_active=false, deactivated_at=now()
6. facility_datasets 갱신 (facility_count, source_file, uploaded_by, updated_at)
7. 결과 집계 (신규/갱신/비활성 시설·기구, 실패 목록)
```

#### 복구·이력

- JSON 재포함 → `is_active=true`, `deactivated_at=NULL`
- `monthly_inspections`, `inspection_ledger_rows`, `monthly_inspection_items` — **변경·삭제 없음**

#### `archived` 데이터셋 (P2)

| 동작 | 허용 |
|------|:----:|
| JSON 업로드 | ✗ |
| 신규 월간 점검 생성 | ✗ (권장) |
| 기존 점검·대장 조회 | ✓ (할당·ADMIN) |

#### service role 사용 현황 (RLS 적용 전 점검)

| 파일 | service role | 조치 |
|------|:------------:|------|
| `settings/user-actions.ts` | Auth 사용자 생성·이메일 | assignments는 **`createClient()`(사용자 JWT)** 권장 |
| `admin/upload/actions.ts` | **미사용** | 사용자 JWT + ADMIN |
| 그 외 Server Actions | **미사용** | RLS 적용 대상 |

service role은 RLS를 우회하므로, datasets/assignments 테이블에는 **사용자 JWT 클라이언트**로 쓰거나 service role 사용 시 **동일 규칙을 코드로 이중 검증**한다.

### 14.9 RLS 적용 순서

1. 앱 레이어 필터·`assertFacilityAccess` 완료
