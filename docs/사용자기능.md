# 사용자별 관리기능

어린이놀이시설 월간 안전점검 대시보드의 **역할(Role)·권한(Permission)·기능**을 코드 기준으로 정리한 문서입니다.

- **권한 단일 소스:** [`src/lib/auth/permissions.ts`](../src/lib/auth/permissions.ts)
- **역할 타입:** [`src/types/inspection.ts`](../src/types/inspection.ts) — `AppRole`
- **역할 저장:** Supabase `inspection_user_roles` 테이블
- **문서 기준일:** 코드베이스 현행 구현 (앱 레벨 + DB RLS)

---

## 1. 개요

### 1.1 시스템 구성

| 계층 | 역할 |
|------|------|
| **앱 (Next.js)** | 페이지 `requirePermission`, 서버 액션 `hasRole`/`hasPermission`, UI 메뉴·버튼 표시 |
| **DB (Supabase RLS)** | `app_private.has_role()` 기반 행 수준 접근 제어 |
| **인증** | Supabase Auth — `(dashboard)/layout.tsx`에서 `requireUser()` |

`middleware.ts`는 없으며, 비로그인 사용자는 `src/lib/supabase/proxy.ts`에서 `/login`으로 리다이렉트됩니다. **역할별 URL 차단은 페이지·액션 단위**로만 적용됩니다.

### 1.2 역할 4종 요약

| 코드 | 한글명 | 사용 가능 기능 수 | 설명 |
|------|--------|:-----------------:|------|
| `ADMIN` | 시스템 관리자 | 15/15 | 전체 시스템 관리 권한 |
| `MANAGER` | 안전관리자 | 12/15 | 시설 관리 및 점검 전 과정 (UI에서 **추천** 역할) |
| `INSPECTOR` | 위탁점검자 | 12/15 | 현장 점검 업무 (위탁점검자 서명 가능) |
| `VIEWER` | 일반 조회자 | 5/15 | 조회 및 대장 출력, 자기 정보만 |

회원가입은 제공하지 않으며, 계정은 시스템 관리자가 생성합니다.
신규 계정 기본 역할: `VIEWER` (Auth 사용자 생성 트리거 / 역할 행 없을 때 fallback).

## 3. 대시보드 세부 기능 분석

로그인 후 대시보드 영역은 `src/app/(dashboard)/` 아래에 있으며, **모든 하위 경로에 로그인 필수** (`requireUser()`).

### 3.1 페이지·경로별 기능

#### `/dashboard` — 대시보드 홈

| 항목 | 내용 |
|------|------|
| **페이지 가드** | 로그인만 (`requireUser`) |
| **주요 기능** | 인사말, 날씨 위젯, 일일 안전 퀴즈, KPI(시설 수·당월 점검 상태), 월별 차트, 최근 점검 5건, Quick Actions |
| **데이터 읽기** | `facilities`, `monthly_inspections` 집계 |
| **역할 제한** | 없음 (Quick Actions의 「새 점검」 링크는 VIEWER에게도 표시 → 목적지에서 차단) |

#### `/facilities` — 시설 목록

| 항목 | 내용 |
|------|------|
| **페이지 가드** | 로그인만 (permission `facility:list`는 정의만, 페이지에서 미호출) |
| **주요 기능** | 시설번호·시설명·주소·설치장소 검색, 페이지네이션, 당월 점검 완료/미완료, 행 액션 「상세」「점검」 |
| **역할 제한** | 「점검」 버튼 UI는 **전 역할에 표시** → `/inspections/new`에서 `inspection:create` 없으면 redirect |

#### `/facilities/[facilityNo]` — 시설 상세

| 항목 | 내용 |
|------|------|
| **페이지 가드** | 로그인만 |
| **주요 기능** | 기본정보, 놀이기구, 정기시설검사, 안전교육, 보험, 관리주체, **월간점검 이력** 탭 (7탭, 조회 위주) |
| **액션** | 「점검 시작」→ `/inspections/new?facilityNo=...` |
| **이력 탭** | 대장 링크, 편집·삭제 버튼 (삭제는 서버에서 권한 검사) |

#### `/inspections/new` — 월간 안전점검 시작

| 항목 | 내용 |
|------|------|
| **페이지 가드** | `requirePermission("inspection:create")` |
| **서버 액션** | `createMonthlyInspection` — `hasRole(["ADMIN","MANAGER","INSPECTOR"])` |
| **주요 기능** | 시설 검색, 작성중(draft) 점검 이어쓰기, 점검월·점검일 선택 후 생성 |
| **비즈니스 규칙** | 동일 `(facility_no, inspection_month)` 존재 시 신규 생성 없이 기존 점검으로 이동; 활성 기구로 `monthly_inspection_items` 자동 생성 |

#### `/inspections/[inspectionId]` — 점검 입력·편집

| 항목 | 내용 |
|------|------|
| **페이지 가드** | `draft` → `inspection:edit` 없으면 `/inspections/history` redirect |
| | `completed`/`locked` → `inspection:edit-completed` 없으면 `/ledger` redirect |
| **서버 액션** | `saveInspectionDraft`, `saveInspectionDraftAndOpenLedger`, `completeInspection` |
| **주요 기능** | 기구별 결과(GOOD/CAUTION/REPAIR/STOP_USE), 비고, 안전관리자·위탁점검자 명/서명, 임시저장, 점검완료, 대장 미리보기 |
| **UI (`InspectionForm`)** | 서명·완료·편집 필드는 permission별 표시/비활성화 |

#### `/inspections/[inspectionId]/ledger` — 안전점검실시대장

| 항목 | 내용 |
|------|------|
| **페이지 가드** | 로그인만 (`ledger:print`는 페이지에서 미검증) |
| **주요 기능** | 연초~점검월 누적 대장, 모바일 요약, 인쇄(PDF), 서명 signed URL |
| **수정 버튼** | `!locked` 이고 (completed면 `edit-completed`, 아니면 `edit`) 일 때만 표시 |

#### `/inspections/history` — 점검이력

| 항목 | 내용 |
|------|------|
| **페이지 가드** | 로그인만 |
| **주요 기능** | 시설번호·점검월·상태 필터, 목록, 대장/편집/삭제 액션 |
| **편집 버튼** | draft → `inspection:edit`, completed → `inspection:edit-completed`, locked → 숨김 |
| **삭제 버튼** | `settings:inspection-history-manage` (UI) |

#### `/settings` — 설정

| 항목 | 내용 |
|------|------|
| **페이지 가드** | 로그인만 |
| **주요 기능** | `settings:view` 있으면 서명 정책 카드(placeholder), 없으면 접근 안내 + 자기 정보 링크 |

#### `/settings/account` — 자기 정보

| 항목 | 내용 |
|------|------|
| **페이지 가드** | `requireUser()` |
| **주요 기능** | 이름·소속·전화번호·비밀번호 변경 (`settings:account` — 전 역할), 이메일은 읽기 전용 |

#### `/settings/users` — 사용자 관리

| 항목 | 내용 |

## 4. CRUD 기능 매핑

### 4.1 시설 (Facility)

| 작업 | 앱 허용 역할 | 구현 경로 | DB RLS |
|------|-------------|-----------|--------|
| **Create** | ADMIN | `/admin/upload` JSON 업로드 | ADMIN만 INSERT |
| **Read** | 전 역할 | `/facilities`, `/facilities/[no]`, 대시보드 | authenticated SELECT |
| **Update** | ADMIN | JSON 업로드(upsert) | ADMIN만 UPDATE |
| **Delete** | — | UI 없음 | — |

관련 엔티티(기구, 법정검사, 안전교육, 보험, 관리주체)도 JSON 업로드 시 **시설별 delete 후 insert** (ADMIN).

### 4.2 월간 점검 (Monthly Inspection)

| 작업 | 앱 허용 역할 | 구현 경로 | DB RLS |
|------|-------------|-----------|--------|
| **Create** | ADMIN, MANAGER, INSPECTOR | `/inspections/new` | operator 3역할 INSERT |
| **Read** | 전 역할 | history, facility 탭, ledger, dashboard | authenticated SELECT |
| **Update (draft)** | ADMIN, MANAGER, INSPECTOR | 점검 폼 임시저장 | operator 3역할 — `draft`, `needs_revision` (`20260526120000`) |
| **Update (completed)** | ADMIN, MANAGER, INSPECTOR | `inspection:edit-completed` | operator 3역할 — `completed` 포함 (`20260526120000`) |
| **Complete** | ADMIN, MANAGER, INSPECTOR | `completeInspection` | status → completed + ledger insert |
| **Delete** | ADMIN, MANAGER, INSPECTOR | history·시설 이력 삭제 | operator 3역할 DELETE (`20260525233000` 마이그레이션) |

점검 상태: `draft` → `completed` → (코드상) `locked`, `needs_revision`.

**점검 항목 (`monthly_inspection_items`)**: 점검 생성 시 기구 목록으로 자동 생성; 저장 시 결과·비고 UPDATE.

**대장 (`inspection_ledger_rows`)**: 완료 시 스냅샷 INSERT; 점검 이력 삭제 시 CASCADE 삭제(operator DELETE). 완료 후 수정 저장 시 refresh(delete+insert) — operator INSERT·DELETE 허용, 부모 `completed` UPDATE 정합화 후 refresh 가능.

### 4.3 서명 (Storage `signatures`)

| 작업 | 앱 | DB RLS |
|------|-----|--------|
| **Upload/Update** | `inspection:sign-safety-manager` 또는 `inspection:sign-consigned-inspector` | bucket 업로드: **owner = auth.uid()** (역할 무관) |
| **Read** | ledger signed URL | authenticated |
| **Delete** | — | ADMIN만 |

### 4.4 사용자 (Auth + `inspection_user_roles`)

| 작업 | 앱 허용 역할 | 구현 | 비고 |
|------|-------------|------|------|
| **Create** | ADMIN | `createUserAction` + service role | 역할 MANAGER/INSPECTOR/VIEWER만 |
| **Read** | ADMIN | `/settings/users` | `SUPABASE_SERVICE_ROLE_KEY` 필요 |
| **Update (역할)** | ADMIN | `updateUserRoleAction` | ADMIN 계정·마지막 ADMIN 변경 불가 |
| **Update (이름/전화/비번)** | 전 역할 | `/settings/account` | 본인만 |
| **Update (이메일)** | ADMIN | `/settings/users` | 관리자 전용 |
| **Delete** | — | UI 없음 | — |

## 5. 역할별 권한 및 기능 비교 (요약)

### 5.1 시스템 관리자 (ADMIN) — 15/15

- 시설 JSON 업로드, 사용자 관리, 설정(서명 정책), 점검 전 과정, 양쪽 서명, 이력 삭제, 대장 출력, 자기 정보
- DB RLS상 **`locked` 점검 UPDATE** 등은 ADMIN만(무제한). 점검 삭제·`draft`/`needs_revision`/`completed` 수정은 operator 3역할과 동일

### 5.2 안전관리자 (MANAGER) — 12/15 — 추천

- 점검 생성·입력·완료·완료 후 수정, 안전관리자 서명, 이력 조회·삭제, 대장, 설정, 자기 정보
- **불가:** JSON 업로드, 위탁점검자 서명, 사용자 관리

### 5.3 위탁점검자 (INSPECTOR) — 12/15

- MANAGER와 유사 + **위탁점검자 서명**
- **불가:** JSON 업로드, 사용자 관리, **설정 화면** (`settings:view`) — 내정보는 `/settings/account`로 접근

### 5.4 일반 조회자 (VIEWER) — 5/15

- 시설 목록·상세, 점검 이력 조회, 대장 출력(인쇄), 자기 정보
- **불가:** 점검 생성·편집·완료·삭제, 서명, JSON 업로드, 사용자 관리, 설정(서명 정책)

## 7. 주요 비즈니스 규칙 및 제한사항

### 7.1 점검 워크플로우

1. 시설·점검월 선택 → `monthly_inspections` (status=`draft`) + 기구별 항목 생성
2. 임시저장: 항목 결과·비고, 역할별 서명 필드
3. 완료: `validateCompletion` — 점검월/일, 항목 존재, 비GOOD 시 비고 필수, **안전관리자명·서명 필수**, 위탁점검자명은 선택
4. 완료 시 `inspection_ledger_rows` 스냅샷 생성 후 대장 페이지로 이동

### 7.2 서명 권한 (UI + 서버)

| 필드 | Permission |
|------|------------|
| 안전관리자명·서명 | `inspection:sign-safety-manager` |
| 위탁점검자명·서명 | `inspection:sign-consigned-inspector` |

MANAGER는 위탁점검자 서명 불가, INSPECTOR는 가능, ADMIN은 둘 다 가능.

### 7.3 사용자 관리

- 생성·역할 변경: **ADMIN만**
- 부여 가능 역할: `MANAGER`, `INSPECTOR`, `VIEWER` (ADMIN은 UI/API에서 생성 불가)
- `ADMIN` 역할 행은 역할 변경·일시 정지 화면에서 수정 불가
- 마지막 ADMIN이 자신을 다른 역할로 바꾸는 것 불가

#### 계정 일시 정지·해제

| 항목 | 내용 |
|------|------|
| **저장** | `inspection_user_roles.status` (`active` \| `suspended`), `suspend_reason`, `suspended_until`, `suspended_at`, `suspended_by` |
| **정지** | ADMIN · `suspendUserAction` — 대상이 ADMIN이 아니고 본인이 아님. 정지 시 전역 세션 revoke |
| **해제** | ADMIN · `reactivateUserAction` — 정지 메타 초기화 |
| **로그인** | 성공 직후 `sync_my_account_status` / `resolveAccountAccess` — 정지면 Sonner 토스트 후 `signOut` |
| **세션** | `proxy.ts`, `requireUser()` — 정지 사용자는 `/login?suspended=1&login=open`으로 이동 |
| **RLS** | `app_private.user_is_active()` — 정지 사용자는 `has_role` 기반 정책 전부 실패. `suspended_until` 만료 시 자동 active |

구현: `src/lib/auth/account-status.ts`, `account-access.ts`, `settings/user-account-actions.ts`

### 7.4 앱 vs DB RLS 불일치 (중요)

대부분의 점검 CRUD는 앱·DB가 일치합니다. **남은 불일치**는 아래와 같습니다.

| 작업 | 앱 (`permissions.ts`) | DB RLS |
|------|----------------------|--------|
| 시설·참조 데이터 쓰기 | ADMIN (`facility:upload`) | **ADMIN만** |

**정합화됨**

| 날짜 | 작업 | 앱·DB |
|------|------|--------|
| 2026-05-25 | 점검 이력 삭제 | ADMIN, MANAGER, INSPECTOR |
| 2026-05-26 | 완료 점검 수정·항목 UPDATE·ledger refresh | ADMIN, MANAGER, INSPECTOR (`draft`/`needs_revision`/`completed`; `locked`는 ADMIN만) |

**역할별 수정 범위 (DB):** operator(MANAGER, INSPECTOR)는 `locked` 상태 점검을 UPDATE할 수 없습니다. 앱에서도 `locked`는 편집 버튼이 숨겨집니다.

증상 예 (남은 불일치): JSON 업로드 등 시설 데이터 쓰기 실패. 점검·ledger 관련 `delete-failed`·저장 실패는 마이그레이션 미적용·`locked` 상태일 때 확인.

### 7.5 UI에서 권한 미검증 노출 (서버에서 차단)

다음 링크/버튼은 **역할과 무관하게 표시**될 수 있으며, 실제 동작은 페이지·액션·RLS에서 막힙니다.

- 대시보드 Quick Actions 「새 점검 시작」
- 시설 목록 「점검」 버튼
- 시설 상세 「점검 시작」
- 시설 상세 월간점검 이력 「편집」「삭제」(삭제는 액션 권한·RLS 이중 검사)

### 7.6 기타

- `completeInspection` 서버 액션은 `inspection:complete` permission 문자열을 **직접 검사하지 않음** (OPERATOR `hasRole` + UI 버튼 숨김에 의존)
- `ledger:print` permission은 정의되어 있으나 **ledger 페이지에서 미검증** (로그인만으로 인쇄 가능)
- Feature flag: 코드에 **없음**
- 점검자 이름 필드(`safety_manager_name` 등)는 **역할명이 아닌** 점검 기록용 텍스트
